## Findings

### Finding 1 — Malformed JSON is compiled as HIGH readiness with no warning (**design question**)

Reproducer:
```
$ printf '{"broken": ' > bad.json
$ aksharamd compile bad.json -o out --json --quiet
{"success": true, "readiness_score": 88, "quality_band": "HIGH", "warning_codes": [], "chunks": 1, ...}
```

`document.md` contains the raw truncated input in a fenced ```` ```json ```` block; no JSON parsing was attempted and no warning surfaces. A CI/CD ingestion gate configured with `--min-readiness-score 70` would silently accept this file.

**Classification**: design question, not a shipping regression. `.json` extension is a hint, not a schema commitment. The lenient plain-text fallback is a deliberate parser choice for robustness. But the readiness signal today implies confidence that the input parsed cleanly, which isn't true.

**Recommended follow-up**: file a roadmap issue proposing that data parsers (`.json`, `.xml`, `.yaml`, `.csv`) surface a `W_PARSE_FALLBACK` informational warning when the primary parser fails and the file is preserved as raw text. Small penalty (5-10 points) so band shifts from HIGH → OK, but no hard block. Out of scope for this validation PR.

### Finding 2 — `validate` CLI does not accept `--json` (**documentation gap**)

```
$ aksharamd validate <file> --help
Options: -v, --verbose
```

The plain output is:
```
Validation passed
```

Documentation and other CLI subcommands lead a reasonable user to expect `--json` (per `compile --json`). Non-blocking; two-line fix: either add `--json` support to `validate` or explicitly document the absence.

### Finding 3 — `show-manifest` requires the nested per-file directory (**minor UX**)

```
$ aksharamd compile file.md -o out/
$ aksharamd show-manifest out/           # ← "No manifest.json found in output dir"
$ aksharamd show-manifest out/file/      # ← works
```

`compile -o <dir>` creates a per-source subdir named after the source stem. `show-manifest <dir>` looks for `manifest.json` directly in `<dir>`, not in the per-source subdir it would find under `<dir>/<stem>/manifest.json`. Non-blocking; users learn quickly, but the error message could point at the nested location.

### Finding 4 — `compiled_at` + timing fields are wall-clock (**intentional, not a defect**)

The following fields differ across identical repeat compile runs:
- `manifest.compiled_at` (ISO 8601 wall clock)
- `manifest.elapsed_seconds`
- `manifest.stage_timings.*` (per-stage timings)
- `document.compiled_at`
- `chunks/<id>.json`'s embedded `compiled_at`

All content-derived identifiers are stable:
- `source_id`, `capture_id`, `document_id`
- Chunk file names (SHA-256 prefix of content)
- Block IDs and `checksum` fields

This is the correct provenance semantics: identifiers derive from content; timestamps derive from wall clock. Not a defect. `docs/IDENTITY_DESIGN.md` should perhaps explicitly note that timestamps are wall-clock and exempt from the determinism contract.
